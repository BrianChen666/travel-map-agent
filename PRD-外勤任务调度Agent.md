## 1. 背景与目标

### 1.1 项目背景
面向广州区域**办公设备售后维保**场景，模拟某服务商为企业客户提供打印机、门禁、POS 机、会议设备、自助终端等设备的上门维修服务。
每天产生大量来自**客户报修表单**与**设备告警**的外勤工单，调度主管需要根据客户地址、预约时间、故障类型、服务等级、工程师技能、备件状态与当前位置安排派单。

**现状痛点：**
- **人工派单依赖经验**：调度质量随主管个人能力波动，难复制、难传承。
- **路线判断与业务规则割裂**：地图工具与派单规则两套系统，靠人脑拼接。
- **急单插入困难**：急单/改约/工程师延误时，全盘重排成本高、易乱。
- **派单原因不可解释**：为什么派给 A 不派给 B，说不清楚，客户/工程师不信服。

### 1.2 问题定义（一句话）
> 用 **LangGraph 编排的 Agent**，把"工单语义理解 + 高德地图计算 + 多目标规则评分"结合，输出**可解释、可验证**的派单决策，并支持急单/异常下的**局部动态重排**。

### 1.3 产品目标

**北极星指标：** 派单一次成功率（派出后未被改派/超时即完成的占比）。

| 类型 | 指标 | 目标值 |
|---|---|---|
| 决策质量 | 派单推荐采纳率（主管认可推荐结果） | ≥ 80% |
| 决策质量 | SLA 达成率（按时到达） | ≥ 90% |
| 可解释 | 派单结果可解释覆盖率（每单都有理由+不推荐原因） | 100% |
| 效率 | 单次派单决策耗时 | ≤ 5s |
| 鲁棒性 | 急单插入/延误后受影响工单范围 | 仅局部重排，已出发任务冻结 |
| 成本 | 跨区成本 / 总路线耗时 | 持续优化 |

### 1.4 非目标（本期不做）
- 不做真实车辆 GPS 实时追踪，位置以工单/模拟数据为准。
- 不做工程师 App 全功能，仅输出调度结果与现场辅助信息。
- 不接入真实 ERP/CRM，先用模拟工单与工程师数据跑通原型。
- 不做大规模车辆路径优化（VRP 求解器），用规则评分近似。

## 2. 目标用户与场景

### 2.1 角色定义

| 角色 | 诉求 | 与 Agent 的关系 |
|---|---|---|
| 调度主管 | 快速、可解释地把工单派给最合适的工程师 | 主要用户：审阅/采纳 Agent 推荐 |
| 外勤工程师 | 合理的任务量、清晰的现场信息 | 接收派单结果 + 现场辅助 |
| 企业客户 | 准时上门、按 SLA 解决 | 间接受益（SLA 达成） |

### 2.2 用户画像（示例）
- **调度主管老张**：每天处理几十张工单，急单一来就手忙脚乱，最怕"派错人 + 解释不清" → 需要"Agent 给推荐 + 理由，我点确认"。
- **工程师小陈**：希望任务不要忽多忽少、地址别太离谱 → 需要"负载均衡 + 地址置信度提示"。

### 2.3 核心场景（覆盖 6 类测试场景）

| 场景 | 说明 | Agent 关键能力 |
|---|---|---|
| 普通报修 | 标准工单，正常派单 | 候选筛选 + 多目标评分 |
| 重点客户急单 | 高 SLA 客户突发，需插入 | 急单插入 + 局部重排 |
| 客户改约 | 时间变更，需重新评估 | 重新评分 + 冲突检测 |
| 工程师延误 | 在途任务超时，影响后续 | 冻结已出发 + 后续重排 |
| 缺少备件 | 候选工程师备件不满足 | 备件满足度过滤/降权 |
| 地址模糊 | 客户地址不规范 | 地址标准化 + 置信度提示 |

### 2.4 核心用户故事（User Story）

| ID | 角色 | 用户故事 | 优先级 |
|---|---|---|---|
| US-01 | 调度主管 | 作为主管，我想上传/接收工单后自动得到推荐工程师，以便快速派单 | P0 |
| US-02 | 调度主管 | 作为主管，我想看到每个推荐的评分理由和不推荐原因，以便信服并决策 | P0 |
| US-03 | 调度主管 | 作为主管，当急单插入时，我想只重排受影响的工单，以便不打乱全局 | P0 |
| US-04 | 调度主管 | 作为主管，我想看到 SLA 风险、天气风险提示，以便提前干预 | P1 |
| US-05 | 工程师 | 作为工程师，我想任务负载被均衡考虑，以便不被过度派单 | P1 |
| US-06 | 主管 | 作为主管，我想查看调度日志，以便复盘和优化规则 | P1 |

### 2.5 用户旅程（急单插入示例）
```
急单进入 → 工单解析(故障/SLA/地址) → 候选工程师筛选
   → 地图特征计算(可达性/耗时/天气) → 多目标评分排序
   → 检测冲突：冻结已出发任务，仅重排受影响后续单
   → 输出推荐+理由+风险 → 主管确认/调整 → 落单 + 写调度日志
```

## 3. 竞品 / 方案对比

| 方案 | 决策方式 | 可解释 | 鲁棒性 | 我们的差异化 |
|---|---|---|---|---|
| 纯人工派单 | 经验 | 因人而异 | 急单易乱 | 我们标准化、可复制 |
| 纯规则引擎 | 硬规则 | 强但僵硬 | 不懂自然语言工单 | 我们加 LLM 理解非结构化工单 |
| 纯 LLM 派单 | 模型直出 | **不可验证** | 易幻觉乱派 | 我们用规则评分，结果可验证 |
| **本方案 Agent** | LLM 理解 + 规则评分 | **每单有理由** | 局部动态重排 | 兼顾理解力与可解释性 |

**差异化锚点：** LLM 负责"理解"，规则负责"决策"——**避免单纯依赖大模型生成不可验证的派单结果**。

## 4. 产品方案

### 4.1 功能架构（MoSCoW 优先级）

```
外勤任务调度 Agent
├── 工单接入与理解
│   ├── 自然语言/表单工单解析       [Must]
│   ├── 故障类型/SLA/优先级识别      [Must]
│   └── 地址标准化 + 置信度          [Must]
├── 地图计算（高德 MCP）
│   ├── 地点检索 / 地理编码          [Must]
│   ├── 路径规划 / 距离测算          [Must]
│   └── 天气查询 → 风险特征          [Should]
├── 派单决策引擎
│   ├── 工程师候选筛选              [Must]
│   ├── 多目标评分排序              [Must]
│   ├── 调度解释生成（理由+不推荐原因）[Must]
│   └── 动态重排（急单/延误/改约）   [Must]
├── 现场辅助
│   ├── 预计到达时间 / 路线提示      [Should]
│   └── 风险提示（天气/跨区/地址）    [Should]
└── 运营与评估
    ├── 调度日志                    [Must]
    ├── 效果评估记录                [Must]
    └── ECharts 可视化看板          [Should]
```

### 4.2 核心功能详述

#### F1 工单解析（LLM 理解层）
- **输入**：自然语言报修 / 设备告警 / 表单工单。
- **输出（结构化）**：客户、地址、设备类型、故障类型、服务等级(SLA)、预约时间、优先级。
- **要求**：解析结果落为结构化字段，供后续规则计算，**不让 LLM 直接决定派给谁**。

#### F2 地图特征计算（高德 MCP）
将地图原始结果转化为**调度特征**：

| 地图能力 | 转化为调度特征 |
|---|---|
| 地理编码 | 地址置信度 |
| 路径规划 / 距离测算 | SLA 可达性、路线耗时 |
| 跨区判断 | 跨区成本 |
| 天气查询 | 天气风险 |

#### F3 多目标评分（规则决策层，核心）
对候选工程师按多维度加权评分排序：

| 评分维度 | 含义 |
|---|---|
| 技能匹配 | 工程师是否会修该设备/故障 |
| 备件满足 | 是否带/可取所需备件 |
| 客户等级 | 重点客户优先 |
| 工程师负载 | 当前任务量，避免过载 |
| 预计到达时间(ETA) | 越快越优，结合 SLA |
| 任务稳定性 | 减少不必要的改派/扰动 |

> 评分公式 `Score = Σ(wᵢ × featureᵢ)`，权重可配置；输出排序 + 每项得分明细。

#### F4 调度解释生成
- 输出"**推荐工程师 + 推荐理由 + 各候选不推荐原因**"，由结构化评分结果生成自然语言说明（LLM 仅做表达，不做决策）。

#### F5 动态重排
- **急单插入 / 工程师延误 / 客户改约**时触发。
- **冻结已出发任务**，仅对**受影响的后续工单**做局部重排，避免全盘扰动。

## 5. 技术方案

### 5.1 系统架构

```
┌──────────────────────────────────────────────────────────────┐
│ 接入层   Web(ECharts看板) ←→ FastAPI 网关                       │
├──────────────────────────────────────────────────────────────┤
│ 编排层   LangGraph Agent 工作流（见 5.2 状态机）                 │
├──────────────────────────────────────────────────────────────┤
│ 能力层   ① 工单解析(大模型API)  ② 高德MCP(地图/天气)             │
│          ③ 多目标评分引擎(规则)  ④ 解释生成(大模型API)            │
├──────────────────────────────────────────────────────────────┤
│ 知识/检索  RAG + Embedding + Vector Store（故障知识/历史工单）    │
├──────────────────────────────────────────────────────────────┤
│ 数据层   SQL(工单/工程师/备件)  ·  Redis(实时状态/路线缓存/锁)    │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 LangGraph Agent 工作流（状态机）

```
[工单输入]
   → (节点1) 解析工单 → 结构化字段
   → (节点2) 地址标准化（高德地理编码）→ 置信度
   → (节点3) 候选工程师筛选（技能/备件/可达硬约束）
   → (节点4) 地图特征计算（路径/ETA/天气/跨区）
   → (节点5) 多目标评分排序
   → (判定) 是否触发动态重排？
        ├─ 是 → (节点6) 冻结已出发 + 局部重排
        └─ 否 → 直接出推荐
   → (节点7) 调度解释生成
   → [输出：推荐+ETA+评分+风险+不推荐原因] → 写调度日志
```

### 5.3 关键技术点
- **Redis 职责**：缓存高德路径/距离结果(降本提速)、维护工程师实时状态、动态重排时的并发锁(防止重复派单)。
- **Vector Store / RAG 职责**：检索历史相似工单、故障处理知识，辅助工单理解与现场辅助(非派单决策)。
- **SQL 职责**：工单、工程师、备件、客户等级等结构化数据的权威来源。
- **ECharts**：派单结果、负载分布、SLA 达成率、路线可视化。

## 6. 关键设计：可解释派单（产品灵魂）

> **核心原则：LLM 负责"理解与表达"，规则负责"决策"。决策必须可复算、可审计。**

### 6.1 为什么不让大模型直接派单？
- 大模型直出派单结果**不可验证**：换次提问可能换个人，无法解释、无法复盘、无法对账。
- 故采用「**LLM 解析 → 规则评分 → LLM 解释**」三段式：决策点全部落在可配置、可复算的规则评分上。

### 6.2 可解释三件套（每单必出）
1. **推荐工程师** + 总分。
2. **评分明细**：各维度得分(技能/备件/ETA/负载…)。
3. **不推荐原因**：每个落选候选为何被排在后面(如"备件不满足""跨区成本高""已超载")。

## 附录 A：术语表
- **LangGraph**：基于图/状态机编排多步骤 Agent 工作流的框架。
- **高德 MCP**：高德地图的 Model Context Protocol 接口，提供地点检索、地理编码、路径规划、天气等。
- **SLA**：服务等级协议，此处指上门/解决时限要求。
- **ETA**：预计到达时间。
- **动态重排**：异常事件下仅对受影响工单重新分配，已出发任务冻结。
- **多目标评分**：按技能/备件/ETA/负载等多维加权排序候选工程师。

## 附录 B：关键决策记录
- *为什么不让大模型直接派单？* —— 直出结果不可验证、不可解释、不可复盘；改为「LLM 解析 → 规则评分 → LLM 解释」，把决策权交给可复算的规则。
- *为什么动态重排要冻结已出发任务？* —— 已在途的工程师再改派成本高且体验差，只重排受影响的后续单，平衡最优性与稳定性。
