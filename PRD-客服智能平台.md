## 1. 背景与目标

### 1.1 项目背景
面向企业客服知识管理场景，当前存在三类痛点：
- **内部坐席培训成本高**：业务规则、话术、工单流程散落在多个文档，新人查找慢、易出错。
- **外部用户自助率低**：FAQ 静态、关键词匹配漏检严重，复杂问法答非所问，人工压力大。
- **知识更新滞后 + 泄密风险**：通用大模型无私域知识且会"幻觉"；若不做隔离，外部用户可能套出内部敏感信息。

### 1.2 问题定义（一句话）
> 用 RAG 技术，为客服场景提供一个**可追溯、可控权限、低幻觉**的智能问答平台，**严格区分内部员工助手与外部用户助手两类使用场景**。

### 1.3 产品目标

**北极星指标：** 问题一次性解决率（不转人工即解决的占比）。

| 类型 | 指标 | 目标值 |
|---|---|---|
| 体验 | 回答准确率 | ≥ 85% |
| 体验 | 幻觉率（答案不忠于知识库） | ≤ 5% |
| 安全 | 内外部越权泄露率 | ≈ 0%（红线） |
| 效率 | 平均响应时延 | ≤ 3s |
| 业务 | 外部用户自助解决率 | 较 FAQ 提升 ≥ 30% |
| 业务 | 内部坐席平均查询耗时 | 下降 ≥ 50% |

### 1.4 非目标（本期不做）
- 不做多模态（图片/语音）问答，仅文本。
- 不做模型微调，先用"通用大模型 API + RAG"验证。
- 不做全渠道接入（微信/电话），先做 Web 端原型。

> 💡 *写"非目标"是 AI PM 的克制力体现——明确边界，避免范围蔓延。*

## 2. 目标用户与场景

### 2.1 双角色定义（本项目核心）

| 维度 | 内部员工助手 | 外部用户助手 |
|---|---|---|
| 用户 | 客服坐席 / 内部员工 | 终端消费者 |
| 典型场景 | 查业务规则、话术、工单流程、内部知识 | 产品咨询、FAQ 解答、基础售后 |
| 核心诉求 | 全、准、快（降培训成本） | 友好、安全、不泄密 |
| 可访问知识 | 内部库 + 公开库 | **仅公开库** |
| 最大风险 | 规则查错 → 答错客户 | 内部敏感信息越权泄露 |
| 语气风格 | 专业、直给 | 礼貌、有同理心 |

### 2.2 用户画像（示例）
- **坐席小李**：入职 2 周，记不住退换货规则，高峰期来不及翻文档 → 需要"问一句立刻给准确答案+出处"。
- **消费者王女士**：想知道"过保了还能修吗"，不想等人工 → 需要"自然语言提问，得到友好、明确的答复"。

### 2.3 核心用户故事（User Story）

| ID | 角色 | 用户故事 | 优先级 |
|---|---|---|---|
| US-01 | 坐席 | 作为坐席，我想用自然语言查业务规则，以便快速准确答复客户 | P0 |
| US-02 | 坐席 | 作为坐席，我想看到答案的知识来源，以便核实可信度 | P0 |
| US-03 | 消费者 | 作为消费者，我想自助查询售后政策，以便不用排队等人工 | P0 |
| US-04 | 消费者 | 当 AI 答不了时，我想被顺畅转接人工，以便问题最终被解决 | P1 |
| US-05 | 管理员 | 作为管理员，我想给文档配置可见范围，以便控制内外部权限 | P0 |
| US-06 | 管理员 | 作为管理员，我想查看问答日志与差评，以便持续优化 | P1 |

### 2.4 用户旅程（外部用户示例）
```
进入入口 → 输入问题 → AI检索+生成 → 看到带出处的答案
   → 满意(解决) / 不满意(点踩) → 置信度低或点踩 → 转人工 → 上下文打包给坐席
```

## 3. 竞品分析（简版）

| 方案 | 知识库模型 | 隔离/安全 | 我们的差异化 |
|---|---|---|---|
| 传统 FAQ 机器人 | 关键词匹配 | 弱 | 我们用 RAG，理解语义、可追溯 |
| 通用大模型直答 | 无私域知识 | 易幻觉/泄密 | 我们做检索约束 + 权限过滤 |
| SaaS 客服平台 | FAQ+文档抽取 | 数据出域 | 我们强调**内外部物理隔离**、可私有化 |

**差异化锚点：** 可追溯（带出处）+ 强权限隔离 + 低幻觉的拒答边界。

## 4. 产品方案

### 4.1 功能架构（MoSCoW 优先级）

```
客服智能平台
├── 问答交互层
│   ├── 多轮对话窗口        [Must]
│   ├── 答案来源引用展示    [Must]
│   ├── 点赞/点踩反馈        [Should]
│   └── 转人工入口          [Must]
├── 知识管理后台
│   ├── 文档上传/解析/切分  [Must]
│   ├── 知识可见范围配置(内/外/角色)  [Must]
│   ├── 知识更新/下架        [Should]
│   └── 问答日志查询        [Should]
├── 检索与生成引擎(RAG)
│   ├── 意图识别与路由(SQL/RAG)  [Must]
│   ├── 混合检索(向量+BM25)  [Must]
│   ├── Rerank 重排          [Should]
│   ├── 权限过滤(ACL)        [Must]
│   └── Prompt 约束 + 拒答兜底  [Must]
└── 评估与运营
    ├── 黄金测试集回归      [Must]
    ├── 指标看板            [Should]
    └── A/B 实验            [Could]
```

### 4.2 核心功能详述

#### F1 智能问答（带来源引用）
- **输入**：用户自然语言问题 + 用户角色（内部/外部）。
- **处理**：意图识别 → 检索 → 权限过滤 → Rerank → Prompt 组装 → LLM 生成。
- **输出**：自然语言答案 + 引用的知识片段出处（文档名/段落）。
- **约束**：答案必须基于检索内容，无依据时触发拒答。

#### F2 权限隔离（项目灵魂，见第 6 章详述）
- 管理员为每个文档/知识块打标签：`可见范围 ∈ {公开, 内部}`、`部门`、`安全等级`。
- 外部用户助手在**检索阶段**即被限制只能命中"公开"知识。

#### F3 意图识别与多路由
- 结构化问题（如"订单 123 物流到哪了"）→ 走 **SQL** 查询业务库。
- 知识型问题（如"过保还能修吗"）→ 走 **RAG** 检索文档库。
- 路由由意图识别模块判定。

#### F4 拒答与转人工
- 检索/生成置信度 < 阈值 → 输出"这个问题我暂时无法准确回答，正在为您转接人工"，并把上下文打包给坐席。

## 5. 技术方案

### 5.1 系统架构（端到端）

```
┌─────────────────────────────────────────────────────────┐
│ 接入层   Web 前端  ←→  FastAPI 网关（鉴权/路由）           │
├─────────────────────────────────────────────────────────┤
│ 编排层   LangChain：意图识别 → 路由 → 检索 → 重排 → 生成    │
├─────────────────────────────────────────────────────────┤
│ 检索层   Embedding 向量检索 + BM25 关键词检索 + Rerank     │
│          └── ACL 权限过滤（基于用户角色 + 文档标签）        │
├─────────────────────────────────────────────────────────┤
│ 生成层   Prompt 组装(上下文+约束) → 大模型 API → 来源标注    │
├─────────────────────────────────────────────────────────┤
│ 数据层   向量库(公开/内部分库)  ·  业务库(SQL)  ·  日志库    │
└─────────────────────────────────────────────────────────┘
```

### 5.2 RAG 数据流水线

| 阶段 | 设计要点 |
|---|---|
| 文档接入 | 支持 docx/pdf/md，解析保留表格与标题结构 |
| 清洗 | 去噪、去重、统一格式 |
| 切分 Chunking | 每块 ~480 token，重叠 100–150，保证块自包含 |
| Embedding 入库 | 入向量库，**公开与内部分 Collection 存储** |
| Query 改写 | 补全指代、拆解复杂问法 |
| 混合检索 | 向量(语义) + BM25(精确命中编号/单号) |
| Rerank | top-50 → 重排取 top-5 |
| 权限过滤 | 按用户角色 + 文档标签做 ACL 过滤 |
| Prompt 组装 | 注入上下文 + 强约束(只依据材料、无依据则拒答) |
| 生成与质检 | LLM 生成 → 来源引用 → 输出前过滤敏感信息 |

## 6. 关键设计：权限隔离（双保险）

> **核心原则：鉴权在后端统一做，算法侧不直接处理鉴权；隔离做在检索层，而非仅靠生成层提示。**

### 6.1 三层防御
1. **库级隔离**：公开知识与内部知识存入**不同 Collection / Partition**，从源头分开。
2. **检索级过滤（主防线）**：每个知识块打 `可见范围/部门/安全等级` 标签；外部用户的检索请求带角色，**在向量检索阶段就过滤掉内部块**——让外部助手"物理上检索不到"内部知识。
3. **生成级兜底（纵深防御）**：Prompt 中追加"禁止输出任何内部敏感信息"约束。注意：这只是补充，**不能单独依赖**，因为仅靠提示词约束一定会有泄露。

### 6.2 越权测试（必做）
- 用外部角色去问内部专属问题（如"内部审批流程是什么"），期望结果：**检索不到 + 拒答**，越权泄露率必须 ≈ 0。

## 7. Prompt 设计（附模板）

```
# 角色
你是企业客服助手，仅服务于【外部用户 / 内部员工】。

# 知识材料
${context}   # 检索并经权限过滤后的知识片段

# 回答要求
1. 只能依据上面的「知识材料」回答，不得编造。
2. 若材料中没有相关信息，回答："抱歉，这个问题我暂时无法准确回答，正在为您转接人工。"
3. 回答末尾标注信息来源（文档名/段落）。
4. 语气：【外部=礼貌友好；内部=专业直给】。
5. 禁止输出任何标注为"内部"的敏感信息。

# 用户问题
${question}
```

## 8. 评估方案（验收命门）

### 8.1 指标体系（检索段 + 生成段）

| 阶段 | 指标 | 含义 | 目标 |
|---|---|---|---|
| 检索 | Recall@k | 该捞的知识是否被捞回 | ≥ 0.9 |
| 检索 | Precision@k / NDCG@k | 相关项是否排前 | 持续优化 |
| 生成 | Faithfulness（幻觉率） | 答案是否忠于上下文 | 幻觉 ≤ 5% |
| 生成 | Answer Correctness | 对照标准答案是否正确 | ≥ 85% |
| 生成 | Answer Relevance | 是否切题 | ≥ 90% |
| 安全 | 越权泄露率 | 外部能否套出内部知识 | ≈ 0 |
| 边界 | 拒答准确率 | 该拒的有没有拒 | ≥ 90% |

### 8.2 黄金测试集（Golden Set）
构建一套覆盖四类场景的问题集 + 标准答案，每次修改切分/模型/Prompt 后跑回归：

| 场景 | 用途 |
|---|---|
| 知识召回 | 验证该捞的没漏 |
| 答案准确性 | 验证答得对、不瞎编 |
| 拒答边界 | 验证没把握时不硬答 |
| 内外部隔离 | 验证外部碰不到内部知识 |

### 8.3 验收标准
- 全部 P0 指标达标，越权泄露率为 0，方可上线。

## 附录 A：术语表
- **RAG**：检索增强生成，先检索知识再让模型生成答案。
- **Embedding**：把文本转成向量以做语义检索。
- **Rerank**：对初步检索结果重排序，提升 top 结果相关性。
- **ACL**：访问控制列表，按角色/标签控制可见范围。
- **Faithfulness**：答案对检索上下文的忠实度（幻觉的反指标）。
